📘 learn-codex

learn-codex — 独立技术教育评审报告

评审对象：learn-codex（用 Python 从零拆解 OpenAI Codex 架构的教学仓库） 对标基准：learn-claude-code（同作者心中的「高 star」质量标杆） 评审方式：通读根 README + 全部 17 章 README + 2 篇深入长文；精读 codexlib.py 与 5 章 code.py；离线运行全部 17 章 --demo（17/17 退出码 0，含 macOS 上真实的内核沙箱拦截）；审计交叉引用 / 链接 / HTML 产物。 立场：挑刺。目标是找出「这个仓库与高 star 之间还差什么」。

一句话结论

vs learn-claude-code，这是一个「思想讲解水准已经追平甚至局部超越标杆、但成色被一批低级链接/编号瑕疵和单语言+无图的工程短板拖住」的项目。 核心教学内容已经是同类里的第一梯队；离「高 star」只差一轮打磨和几项分发层面的投入（多语言、图、可发现性）。

总分：82 / 100。

各维度评分

1. 思想清晰度 —— 9.5 / 10

这是本仓库最强的一项，已经超过标杆的平均水准。每章开头的「先把思想说透」不是术语清单，而是真正的「推理 + 类比」教学：

  • s03 apply_patch 用「让粗心朋友夹书签：说『第 213 页』会翻错，说『夹在「从前有座山」和「山里有座庙」中间』一找一个准」解释为什么补丁靠上下文锚定而非行号（s03 README 第 26 行）——这是把「LLM 数不准行号」这个抽象事实变得肉眼可懂的范例。
  • s05 sandbox 的三层递进（「门里看门看不住」→「把执法点搬到进程够不着的内核」→「默认全拒、失败方向是安全的」）配合「门卫会被花言巧语骗过，物理门锁不会」（s05 第 18 行）——把「应用层挡 vs 内核层关」讲透。
  • s10 SQ/EQ 从一个具体的「做不到」出发（同步函数调用里无法在「想跑」和「跑了」之间插入审批），一步步逼出双队列，再用「门铃 + 对讲机 vs 自动门」收束（s10 第 74 行）。
  • s02 的「电话簿 / 公司前台转接」、s06 的「法律分层：宪法→省条例→市细则」、s09 的「麦当劳和星巴克的点单格式不同」、s15 的「USB-C 把 N×M 塌缩成 N+M」——几乎每章都有一个贴切且自洽的核心类比。

「走一遍」环节同样扎实：它用的是 --demo 真实跑出来的数据（我逐一核对过，例如 s16 的 provenance 表、s03 的 poem.txt 前后内容、s05 的 Operation not permitted，README 里写的和实际输出完全一致）。

扣 0.5 的唯一理由：极少数章节（s11、s15）「先把思想说透」偏长，三个道理之间略有重复，初学者可能在进入「问题」前就已疲劳。

2. 完整性与覆盖 —— 8 / 10

17 章覆盖了 Codex 区别于 Claude Code 的全部招牌设计：apply_patch、内核沙箱、审批策略、AGENTS.md、Rollout、Responses API、SQ/EQ、多前端、Guardian、MCP 双向、Config/Profile。选题精准，每个 ⭐ 章都打在 Codex 真正独特的地方。覆盖面对「理解 Codex 架构」这个目标是充分的。

但有真实的覆盖缺口：

  • 多智能体 / 子代理只活在长文里，没有章节。 这是最大的结构性缺口。agent-graph-store、agent-identity、InterAgentCommunication、Review 模式、cloud-tasks 这些 Codex 极有特色的东西，只在 docs/subagent-multiagent-cc-vs-codex.md 里讲，没有对应的 code.py 可跑章节。标杆 learn-claude-code 专门有 s15/s16/s17 三章讲 agent teams——本仓库把对标内容降级成了一篇阅读材料。对一个想「理解 Codex」的人，这是缺了一块可动手的拼图。
  • 错误恢复 / 重试 没有独立成章（标杆有 s11 Error Recovery）。Codex 的 responses_retry.rs、粘性路由、reactive 压缩重试散落在 s07/s09 的 <details> 里，没有集中讲。
  • 没有一处总览图/索引把「真源码 crate ↔ 章节」对应起来（s17 的表是「流水线工位 ↔ 章节」，但读者想反查「apply-patch crate 在哪章」时没有入口）。

冗余方面基本没有问题：s02 和 s12 都讲「加工具循环不变」，但 s12 明确把重心移到「可观测事件」，不算重复。

3. 递进与教学法 —— 8.5 / 10

6 部分的编排（给手+缰绳 → 脑子 → 引擎盖 → 前端扩展 → 进阶 → 收尾）是合理的，且有意识地借鉴了标杆的「能力→控制→由内向外→集成→综合」哲学（根 README 第 77 行明说）。难度大体单调爬升，章与章之间有真实的「钩子」（每章结尾「下一站」+ 开头「承接上一章」），s17 的「整门课其实只是一条流水线」收尾非常有力——把 16 章重新组织成「请求穿过一道道关卡，模型只占其中一步」，是点睛之笔。

但递进有两处可议：

  • s07（上下文压缩，标为 ≈）夹在两个 ⭐ 重章 s06 和 s08 之间，且它依赖的 s09（Responses API）/ rollout 概念其实排在它后面。 s07 里大量出现「reasoning 占预算」「服务端压缩」这些要到 s08/s09 才讲清的概念。把「模型怎么被调用」（s09）放在「上下文怎么被压」（s07）之后，是一个可争议的顺序——读者在 s07 会遇到尚未建立的概念。
  • Part 3「掀开引擎盖」（s09 协议 / s10 架构）位置偏后。 Responses API 的 item 形状其实从 s01 就在用（function_call / function_call_output），却到第 9 章才解释。前 8 章一直让读者「先用着、第 9 章再懂」，虽然每章都注明「详见 s09」，但对追求「先理解再使用」的初学者是一种持续的小欠账。

4. 可跑性与忠于源码 —— 9 / 10

这是仅次于「思想清晰度」的强项，也是本仓库最硬的信誉资产。

  • 17/17 章 --demo 全部退出码 0，全程离线、无需 key。 mock 后端设计干净（codexlib.py 的双后端抽象合理）。
  • s05 在 macOS 上真实触发内核沙箱：demo 实际跑出 touch: /Users/.../codex_escape_test.txt: Operation not permitted 且事后断言文件不存在。这不是「假装」——它真的调 /usr/bin/sandbox-exec、真的被内核拒绝。对一个教学仓库，这种「真的能拦住」极大增强可信度。
  • 源码引用精确到文件+行号，且我抽查属实：protocol.rs:760 的 AskForApproval 枚举、seatbelt.rs:602 的 create_seatbelt_command_args、GuardianRiskLevel（approvals.rs:85）、MAX_CONSECUTIVE_GUARDIAN_DENIALS_PER_TURN = 3（s14 熔断器）、DIRECT_MCP_TOOL_EXPOSURE_THRESHOLD = 100（s15）……这些不是泛泛而谈，是真的去读了 codex-rs。「教学版 vs 真源码」的 <details> 对照表是本仓库相对一切同类「换皮教程」的核心差异化。
  • 代码与 README 一致：我核对了 s03（_find_block 的 old/new 列表、demo 补丁）、s10（decision = yield ev(...) 的生成器机制），README 描述的算法和代码逐行吻合，没有发现「README 说一套、代码做另一套」的情况。

扣 1 分的真实理由：

  • s05 的 demo 不自清理——它留下 sandbox_demo/ 目录（README 第 178/197 行让用户手动 rm -rf sandbox_demo），而其余 16 章都用 _demo_workspace/ 并自动 rmtree。这是一处可见的不一致（我在评审中手动清掉了它）。
  • 忠实度的「天花板」是单文件简化，这本身没问题且被诚实标注；但 s03 的 _find_block 只做精确匹配，README 在正文里说「即便数错行号也能贴上」，严格说精确匹配对缩进误差并不鲁棒（README 在 <details> 里有补充说明 seek_sequence，但正文的措辞略微夸大了教学版的鲁棒性）。

5. 🆚 对比深度 —— 9 / 10

对比是本仓库的灵魂，也做得相当好。根 README 的 🆚 总表（10 行，每行带「为什么不同」）+「把为什么收成一句话」（Codex 押自主 / CC 押协作，且模型差异决定工具差异）立住了贯穿全书的主线，每章的「🆚 与 Claude Code 的不同」都把这条主线落到具体机制上，且不回避「这一章两家其实差不多」（s01/s02/s07/s12/s13 诚实地标 ≈，并解释「趋同恰恰说明这是通用最优解」）——这种克制让对比可信，不是为了对比而强行制造差异。

两篇长文（context、subagent/multiagent）的对比深度尤其突出：context 篇从「上下文窗口是什么」讲到 prefix-cache 哲学、服务端 vs 客户端压缩、rollout 解耦，且双向都给了真源码依据（CC 侧引 compact.ts/autoCompact.ts，Codex 侧引 compact.rs/compact_remote.rs）。

扣 1 分：CC 侧的论据强度不对称。Codex 侧处处有 codex-rs 的文件+行号；CC 侧则是「learn-claude-code 对其源码的剖析」——是二手的、且 CC 闭源无法独立验证。长文里若干关于 CC 的精确数字（「保头 3 条」「5 文件×5K/50K 预算」「15 种消息类型」「轮询 Lead 1s / 队友 500ms」）读起来很确定，但读者无从核实，存在「把对二手材料的转述呈现得过于确凿」的风险。至少应注明这些 CC 细节的确定性低于 Codex 侧。

6. 一致性与打磨 —— 6 / 10

这是拉低总分的主要短板，且几乎全部是「廉价可修」的低级瑕疵——正因为可修，它们才尤其刺眼。

确凿的问题（均经 grep + 读 HTML 产物核实）：

  1. 大量「引用式链接」从未定义，在渲染后变成肉眼可见的裸文本 [s04]。 s14 的 index.html 里字面出现 9 次 [s04]、2 次 [s05]、各 1 次 [s10]/[s15]——它们不是可点链接，而是噪声文本。受影响章节：s14（14 处）、s04（9 处）、s13（8 处）。例：s13 第 164 行「与审批策略（[s04]）/ Guardian（[s14]）/ 事件协议（[s10]）同一套」——三个本应是链接的标记全裸露。对照之下 s03、s01 等章节正确地在文末定义了 [s04]: ../s04_approval/。这是「有的章定义了、有的章忘了定义」的不一致，是整个评审里性价比最高的待修项。
  1. 两篇长文用了过时的章节编号，会把读者导向错误章节。 docs/context-cc-vs-codex.md 第 38-43、276 行写「s06 Responses API」「s08 AGENTS.md」「s09 上下文压缩」「s10 Rollout」「s06/s07 协议」——但真实映射是 Responses=s09、AGENTS=s06、压缩=s07、Rollout=s08。即长文里每一个章节指引都是错的（疑似沿用了重排前的旧编号）。读者按图索骥会全部走错。
  1. 线性阅读链有断点。 根 README 宣称按 s01→s17 顺序读，但 s08_rollout 的「下一站」直接跳到 s11（跳过了紧随其后的 s09 Responses、s10 SQ/EQ）。s08 第 252 行：「下一站 [s11 前端：TUI + exec]」。这与「难度单调爬升、逐章承接」的承诺矛盾，也和其它章规规矩矩指向 n+1 的做法不一致。
  1. forward-link 风格不统一：s07 的前向引用藏在「小结」里（指向 s08），而非像其它章那样有独立的「下一站」收尾句——小瑕疵，但破坏了「每章结构同构」的观感。

正面的一面：HTML 产物本身质量不错（18 个 index.html，13–34 KB，表格/代码块/<details> 折叠/引用式链接都正确渲染，build_html.py 是零依赖纯 Python 解析器，本身也是个小教学样本）；术语、格言体例、文件命名（sNN_topic/）高度统一；正文没有残留 TODO/占位（grep 出的几处都是合法的示例文本或对 CC 占位符的准确描述）。问题不在「写得乱」，而在「最后一公里的链接/编号校对没做」。

7. 独特价值 / 会不会有人 star —— 7.5 / 10

独特价值是真实存在的，而且不小： 目前公开生态里，「用可跑的 Python 从零讲清 OpenAI Codex（Rust）内部架构、且每章对照真源码行号、还逐章和 Claude Code 对比」——这个组合几乎没有第二家。learn-claude-code 已证明「从零复刻 + 讲思想」这个品类能拿高 star；本仓库是该品类在 Codex 方向的、质量对标的延伸，且「双系统对比」是它超出单纯复刻的增量价值。gpt-5-codex / Codex 的关注度在涨，选题踩在风口上。

但「会不会真的 star 起来」目前被几个分发层面的短板压制：

  • 只有中文。 标杆是中/英/日三语。Codex 的受众高度国际化，无英文 README 直接砍掉绝大部分潜在 star 来源。这是从「好仓库」到「高 star 仓库」之间最大的单点障碍。
  • 几乎没有图。 标杆用 Mermaid 流程图 + SVG。本仓库通篇是 ASCII 框图——清晰，但在 GitHub 页面上「第一眼吸引力」远不如渲染出的彩图，README 的「截图友好度」低，不利于社交传播。
  • 可发现性弱：根 README 没有 badge、没有一句话英文 tagline、没有指向「在线可读 HTML」的托管链接（HTML 是本地生成的，没部署）。第 6 项的裸 [s04] 噪声也会让第一次点进章节的人觉得「未完成」。

换句话说：内容配得上高 star，包装还停在「自用」阶段。

Top 3 强项（具体）

  1. 「先把思想说透」的类比教学是同类天花板。 apply_patch 的「书签 vs 页码」、sandbox 的「门卫会被骗、门锁不会」、SQ/EQ 的「门铃对讲机 vs 自动门」、MCP 的「N×M 塌缩成 N+M」——几乎每章都有一个能让近乎零基础者「啪一下想通」的核心比喻，而非术语堆砌。这是 learn-claude-code 的精髓，本仓库不仅学到了，局部还更细腻。
  1. 「真的能跑 + 真的对照源码」双重落地。 17/17 demo 离线退出码 0，s05 在真机上被内核当场拒绝越界写；每章 <details> 把教学版精确对到 codex-rs 的文件名+行号+常量名（AskForApproval/GuardianRiskLevel/MAX_CONSECUTIVE_GUARDIAN_DENIALS_PER_TURN=3/DIRECT_MCP_TOOL_EXPOSURE_THRESHOLD=100），抽查全部属实。这把它和「凭印象讲 Codex」的二手教程彻底拉开。
  1. 贯穿全书、且诚实的对比主线。 「Codex 押自主（内核沙箱/可审计 rollout/无头友好）、CC 押协作（交互审批/应用层把关）；而工具差异源于模型训练差异」这条线，从根 README 到每一章「🆚」再到两篇长文一以贯之，且在该说「两家其实差不多」时坦然标 ≈ 并解释趋同——克制让对比可信。
